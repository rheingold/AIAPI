This document describes how security filters are evaluated in the AI UI Automation system.\n\n## Filter Evaluation Principles\n\n### 1. DENY Wins\n- If any filter rule results in DENY, the operation is blocked\n- DENY takes precedence over ALLOW\n- No further evaluation occurs after a DENY\n\n### 2. Default DENY\n- If no explicit ALLOW rule matches, operation is denied\n- System operates on a \"whitelist\" approach\n- All operations must be explicitly permitted\n\n### 3. Evaluation Order\n\nFilters are evaluated in the following order:\n\n1. **Process Name Filters** (Highest Priority)\n   - System process protection (explorer, winlogon, csrss, etc.)\n   - Critical service protection (services, svchost, lsass)\n   - If process matches deny list → DENY immediately\n\n2. **Command Type Filters** (Medium Priority)\n   - Read operations: QUERYTREE, READ, LISTWINDOWS, GETPROVIDERS\n   - Modification operations: SET\n   - UI Interaction: CLICKID, CLICKNAME, CLICK, SENDKEYS\n   - Process Control: LAUNCH, KILL\n\n3. **Parameter Filters** (Lower Priority)\n   - Content-based filtering (keywords, patterns)\n   - Path restrictions\n   - Value validation\n\n4. **Context Filters** (Lowest Priority)\n   - Time-based restrictions\n   - User session validation\n   - Environment checks\n\n## Security Filter Configuration\n\n### Process Name Rules\n```json\n{\n  \"processFilters\": {\n    \"deny\": [\n      \"explorer\", \"winlogon\", \"csrss\", \"lsass\", \n      \"services\", \"svchost\", \"dwm\", \"wininit\"\n    ],\n    \"allow\": [\n      \"calc\", \"notepad\", \"chrome\", \"firefox\",\n      \"excel\", \"word\", \"powerpoint\"\n    ]\n  }\n}\n```\n\n### Command Type Rules\n```json\n{\n  \"commandFilters\": {\n    \"readOnly\": {\n      \"commands\": [\"QUERYTREE\", \"READ\", \"LISTWINDOWS\", \"GETPROVIDERS\"],\n      \"action\": \"ALLOW\"\n    },\n    \"modification\": {\n      \"commands\": [\"SET\"],\n      \"action\": \"REQUIRE_APPROVAL\"\n    },\n    \"interaction\": {\n      \"commands\": [\"CLICKID\", \"CLICKNAME\", \"CLICK\", \"SENDKEYS\"],\n      \"action\": \"ALLOW\",\n      \"restrictions\": {\n        \"maxFrequency\": \"10/minute\"\n      }\n    },\n    \"processControl\": {\n      \"commands\": [\"LAUNCH\", \"KILL\"],\n      \"action\": \"REQUIRE_APPROVAL\"\n    }\n  }\n}\n```\n\n## Filter Evaluation Examples\n\n### Example 1: Safe Read Operation\n- Command: `QUERYTREE`\n- Process: `Calculator`\n- Parameter: `*`\n- **Result**: ALLOW (read operation on safe process)\n\n### Example 2: System Process Interaction\n- Command: `CLICKID`\n- Process: `explorer`\n- Parameter: `closeButton`\n- **Result**: DENY (system process protection)\n\n### Example 3: Dangerous Process Control\n- Command: `KILL`\n- Process: `winlogon`\n- Parameter: `*`\n- **Result**: DENY (system process + dangerous command)\n\n### Example 4: Safe Application Interaction\n- Command: `SET`\n- Process: `notepad`\n- Parameter: `text=Hello World`\n- **Result**: ALLOW (safe process + normal operation)\n\n## Implementation Notes\n\n### Filter Chain\n1. Load security configuration from `security/config.json`\n2. Parse command into: processName, commandType, parameter\n3. Apply filters in order (Process → Command → Parameter → Context)\n4. Return first DENY or final ALLOW/DENY result\n\n### Error Handling\n- Configuration load failure → Default DENY all\n- Invalid command format → DENY\n- Missing security context → DENY\n- Filter evaluation error → DENY\n\n### Logging\n- All DENY decisions are logged with reason\n- ALLOW decisions logged in debug mode\n- Filter evaluation time tracked\n- Security events sent to dashboard\n\n## Command Mapping Reference\n\n| MCP Method | KeyWin Command | Risk Level | Default Action |\n|------------|----------------|------------|----------------|\n| queryTree | {QUERYTREE} | 🟢 Low | ALLOW |\n| readProperty | {read} | 🟢 Low | ALLOW |\n| listWindows | {LISTWINDOWS} | 🟢 Low | ALLOW |\n| getProviders | {GETPROVIDERS} | 🟢 Low | ALLOW |\n| setProperty | {SET:value} | 🟡 Medium | ALLOW |\n| clickElement (ID) | {CLICKID:id} | 🔴 High | ALLOW |\n| clickElement (Name) | {CLICKNAME:name} | 🔴 High | ALLOW |\n| clickElement (Coords) | {CLICK:x,y} | 🔴 High | ALLOW |\n| sendText | SENDKEYS | 🔴 High | ALLOW |\n| launchProcess | {LAUNCH:exe} | ⛔ Critical | REQUIRE_APPROVAL |\n| terminateProcess | {KILL} | ⛔ Critical | REQUIRE_APPROVAL |\n\n## Security Best Practices\n\n1. **Principle of Least Privilege**: Grant minimum required permissions\n2. **Defense in Depth**: Multiple filter layers provide redundancy\n3. **Explicit Allow**: All permissions must be explicitly granted\n4. **Audit Trail**: All security decisions are logged and traceable\n5. **Regular Review**: Security filters should be reviewed and updated regularly\n
